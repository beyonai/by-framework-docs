架构演进：旧架构的“三座大山”

传统分布式 Agent 调度存在严重设计缺陷，亟需底层基座的彻底革新：

1. 全链路同步阻塞

前端请求与后端推理强绑定，高并发下业务网关线程池迅速耗尽，引发系统雪崩。

2. 无法支撑超长任务

Agent 深度推理动辄耗时数分钟，传统同步 HTTP 请求面临致命的网关超时截断问题。

3. 网络耦合与寻址地狱

节点间网络强耦合，需硬编码目标 Worker IP。缺乏服务发现机制，无法弹性扩缩容。

为了彻底粉碎这些痛点，By-framework 应运而生。

底层基座：By-framework 引擎架构解析

分布式高性能调度引擎

By-framework 是构建整个 Agent 生态的基础设施，采用高度解耦的事件驱动架构设计。

Redis Streams 中枢 事件驱动解耦 跨层级隔离
  • 多语言全覆盖 SDK：提供 Python、Java、TypeScript 三种语言 SDK 支持，打破业务开发语言壁垒。
  • 清晰的架构分层：接入层（GatewayClient）拦截请求，执行层（Worker）处理计算，消费层渲染结果。

By-framework 核心 (1/3)：边界解耦与协议转换

化解同步与异步的并发阻抗

前端业务系统（如 HTTP/WebSocket）的请求通常是同步且短时的，而 Agent 的大模型推理则是异步且长耗时的。

GatewayClient 在这里充当了完美的“隔离带”。

  • 流量拦截：拦截前端同步请求，包装为标准的任务指令。
  • 协议转换：转化为异步 AskAgentCommand 压入 Redis 消息队列。
  • 极速释放：业务后端即刻释放线程，高并发下绝不会发生阻塞挂起。

By-framework 核心 (2/3)：调度与智能路由

作为分布式流量集线器

彻底告别传统的、需要硬编码 IP 端口的 RPC 调用。Gateway 提供了一种更为优雅的服务发现机制。

网关层自动接管了复杂的任务寻址工作：

  • 类型声明：调用方只需在指令中声明 target_agent_type 即可。
  • 智能分发：依托 Redis Consumer Group，系统自动拉取并分发任务。
  • 透明路由：任务精准路由给在线、空闲且匹配该类型的分布式 GatewayWorker，天然实现负载均衡。

By-framework 核心 (3/3)：双流全景管控

接管 Agent 分布式生命周期

网关引擎不只负责发包，它分离了控制数据与业务数据，实现了从头到尾的闭环管控：

控制流 (Control Plane)

自上而下。负责下发任务指令、监控 Worker 心跳状态，并在异常触发时发送 CancelTaskCommand，安全中断远端协程。

数据流 (Data Plane)

自下而上。Agent 产生的流式字符（如 emit_chunk）和状态转移，统一封装为流式报文，跨节点推回消费端。

实战基石：By-framework Worker SDK 核心规范

为了确保分布式系统的高可用，Code Agent 在构建数字员工时，必须严格遵循 SDK 定义的工程范式与事件机制。

强制工程目录结构

构建 Gateway Worker 时，必须遵循以下结构，不可变更：

my-worker/
├── src/
├── worker.ts # 继承 GatewayWorker (核心逻辑)
├── run.ts # 独立进程启动 (WorkerRunner)
└── index.ts # 必须 export default runWorker
├── dist/ # tsup 编译输出产物
├── package.json
└── tsup.config.ts

主动拉取模型 (Pull Model)

Worker 主动通过 Redis Stream 拉取任务，而非被动接收 HTTP 请求。这种设计天然支持无状态集群扩展，完美应对大流量洪峰。

全能型 AgentContext

执行上下文不仅提供 emitChunk 进行 SSE 流式响应，还内置了 checkCancelled()，可在长耗时任务中实现一键级联中断，防止算力泄漏。

标准化状态机流转

接管生命周期，严格遵循：
QUEUED → STARTING → COMPLETED / FAILED / CANCELLED

实战环节 ①：Code Agent 如何写代码？

依靠 by-framework 技能，Code Agent 会自动为您生成符合 Gateway 规范的强制三文件结构，无需人工介入模板编写。

src/worker.ts
// 核心业务逻辑实现

export class MyWorker extends GatewayWorker {
  getAgentTypes() {
    return ['my-agent'];
  }

  async processCommand(cmd, ctx) {
    for (const char of text) {
      await ctx.emitChunk({ content: char });
    }
    return { status: 'done' };
  }
}

实战环节 ②：Code Agent 如何自动测试？

Code Agent 熟知框架测试规范。写完代码后，它会利用底层的控制流和数据流机制，启动**无人工干预**的本地自测与逻辑纠错闭环。

1. 启动守护进程

npm run dev (或 tsx src/run.ts)
启动 WorkerRunner 监听 Redis。

2. 开启数据流订阅

npx tsx reference/verify-subscribe.ts
在另一终端监听流式响应，防遗漏。

3. 模拟网关发包

npx tsx reference/send-test.ts
注入测试数据，触发业务逻辑。

4. 强制结构断言

npx tsx src/verify-index.ts
强校验 `runWorker` 导出签名合法性。

实战环节 ③：编译、上传与注册

代码测通后，Code Agent 将执行标准化的发布流线：将 TS 源码编译打包，利用工具注册资源 ID，并自动推送至 MinIO 远端对象存储。

1. 本地构建 (tsup)

依据 tsup.config.ts 规范，将源码打包为 dist/ 产物。

$ npx tsup

2. 平台注册

调用 register_agent 工具，获取唯一的 resourceId。

API: register_agent()

3. 推送 MinIO

使用 mc cp 将 dist/* 推送至以 resourceId 命名的远端路径。

$ mc cp -r dist/* myminio/...
最后通过 query_agent_detail 工具进行结果确权，数字员工正式诞生！
